1) How a Web Request Works

Every AI product is built on web infrastructure. Before understanding AI systems, PMs must know how data moves from a user's device through servers and back. This lifecycle — browser, edge, server, database — is the same path your model API calls travel on every request.

  • A browser initiates HTTP requests; servers respond with data — every LLM API call follows this identical round-trip pattern
  • CDNs cache static assets at geographic edge nodes, reducing latency — critical for global fintech products serving customers across regions
  • Databases persist state between requests — your model context history, audit logs, and customer ledger all live in a database layer
  • In-memory caches (Redis) hold hot data to avoid slow DB reads — PMs track cache-hit rate because it directly controls p95 latency

2) API Gateway & Microservices

Modern fintech products are composed of small, independently deployable services connected by APIs. The gateway is the front door: it enforces auth, rate limits, and routing before any domain service sees the request. PMs must understand this layer because it governs what your AI feature can and cannot do — and who owns the data access.

  • The gateway enforces auth and rate limits before any service logic runs — compliance guardrails often live here as middleware
  • Microservices own single domains: auth, KYC, credit policy, payments — AI orchestration must call each with scoped permissions
  • Service-to-service hops add latency — a six-step AI workflow touching three services must budget each call's p95 latency contribution
  • API contracts define what data AI can access — PMs negotiate field-level data access with platform teams before writing the feature spec

PM Takeaway: Every AI action that touches a customer's account must pass through this stack. "Can we just call the model directly?" — no. Map your AI feature's service dependencies before writing the spec, or you will discover permission blockers in sprint 3.

3) LLM Mechanics: Tokens, Transformers & Embeddings

Language models convert text to tokens, process them through a transformer, and produce probability distributions over possible next tokens. Embeddings are a parallel path: text becomes a fixed-size vector used for semantic search and retrieval. PMs who understand this can make informed decisions about context windows, inference costs, and retrieval quality.

  • Tokens are sub-word units (~0.75 words each) — a 200K token context window holds roughly 150,000 words or a 500-page policy document
  • Transformers use self-attention to weigh every token against every other — this is why long contexts cost more (attention scales quadratically)
  • Embeddings encode semantic meaning as vectors — "similar" text clusters together geometrically, powering semantic search in RAG pipelines
  • Token cost is your primary AI unit-economics lever — every extra word in your system prompt is billable input on every single request

PM Takeaway: Context window × token price = your AI cost ceiling. Every design decision — how much policy text you retrieve, how long your system prompt is, whether you include conversation history — is simultaneously a cost and latency decision. The PM who owns the token budget owns the P&L.

4) Retrieval-Augmented Generation (RAG)

RAG solves the core fintech AI problem: models don't know your latest credit policy, customer ledger, or KYC guidelines. RAG retrieves the right context at query time and injects it into the prompt, grounding every answer in real, current data. This is the architecture behind every policy assistant, credit copilot, and compliance Q&A tool in financial services.

  • The ingestion pipeline runs offline: documents are chunked, embedded, and stored in a vector store — this is not in the query latency path
  • The query pipeline runs in real time: the user's query is embedded, matched against stored vectors, and top-k chunks are injected into the prompt
  • Reranking a cross-encoder model re-scores retrieved chunks for relevance — it adds ~30–80ms but significantly improves precision when policy accuracy is a compliance requirement
  • Data freshness is a risk: stale chunks produce outdated policy answers — your re-ingestion schedule is a product commitment, not an engineering detail

PM Takeaway: In fintech, RAG quality equals compliance quality. Your cited-answer coverage rate, chunk freshness SLA, and retrieval precision are product KPIs — not just engineering metrics. Own the data pipeline that feeds RAG, or you will discover stale policies in a customer complaint.

5) Agentic AI Systems

Agents extend LLMs with the ability to plan, use tools, and act across multiple steps autonomously. A fintech agent might check a customer's ledger, verify a policy, call a payment API, and produce a compliance log — without a human in the loop. PMs must understand the reliability math: each step introduces failure probability, and agents multiply these risks. Six steps at 85% reliability each yields only 37% end-to-end success (0.85⁶ ≈ 0.377).

  • The ReAct loop (Reason + Act) is the core agent pattern: the LLM observes state, reasons about what tool to call, acts, observes the result, and repeats until the goal is met
  • Tool calls are deterministic wrappers around external systems — idempotency and maker-checker approvals are non-negotiable when tools execute real financial actions
  • Memory maintains state across reasoning steps — short-term context in the prompt, long-term facts in a database the agent can query
  • Human-in-the-loop checkpoints are a PM decision: define exactly where the agent must pause for human approval before executing irreversible actions

PM Takeaway: Agent reliability degrades with every step. Define your maximum autonomous action boundary before engineering does. In a payments or credit agent, the question is not "can the agent do this?" — it is "should it, without a human checkpoint?" Write this boundary into the acceptance criteria.

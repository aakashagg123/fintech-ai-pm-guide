1) How a Web Request Works

Every AI product in an NBFC is built on web infrastructure. Before understanding AI systems, PMs must know how data moves from a customer's device through servers and back — whether that's a UPI dispute query hitting NPCI rails or a credit decision traversing CBS. This lifecycle — mobile app, CDN, app server, Core Banking System — is the same path your LLM API calls travel on every loan origination or servicing request.

  • A mobile app initiates HTTP requests; NPCI/CBS servers respond with data — every LLM API call for a credit decision or UPI dispute follows this identical round-trip pattern
  • CDNs cache static assets at geographic edge nodes close to Indian cities — critical for Bharat-scale NBFC products serving Tier 2/3 markets on Jio 4G networks
  • Core Banking Systems (CBS) persist the source of truth between requests — your model context, audit logs, and customer loan ledger all ultimately flow through the CBS layer
  • In-memory caches (Redis) hold hot data to avoid slow CBS reads — PMs track cache-hit rate because it directly controls p95 latency on UPI and NBFC servicing queries

2) API Gateway & Microservices

NBFC products are composed of small, independently deployable services connected by APIs — from NPCI payment switches to CBS modules to Account Aggregator consent layers. The gateway is the front door: it enforces Aadhaar-linked auth, rate limits, and routing before any domain service sees the request. PMs must understand this layer because it governs what your AI feature can and cannot do — and who owns the data access under RBI's AA Master Direction.

  • The gateway enforces Aadhaar eKYC authentication and PMLA rate limits before any service logic runs — RBI compliance middleware often lives here as a mandatory pre-condition
  • Microservices own single domains: auth, CKYC, NBFC credit policy, NPCI payments — AI orchestration must call each with scoped permissions per RBI data governance requirements
  • Service-to-service hops add latency — an AA data fetch triggering CIBIL lookup, policy retrieval, and model inference must budget each call's p95 contribution to stay within NPCI's session timeout
  • AA API contracts define what financial data AI can access — PMs negotiate field-level AA consent scope with platform teams before writing the feature spec, as purpose-creep violates RBI AA Master Direction

Microservice anti-patterns in NBFC:
  • Distributed monolith: 20 microservices that must all be deployed together because they share a database or have synchronous circular dependencies. You have the operational complexity of microservices with none of the benefits. Common in NBFCs that "migrated to microservices" without redesigning domain boundaries
  • Chatty gateway: a single user request triggers 8+ inter-service calls (auth → CKYC → bureau → AA → BRE → scorecard → CBS → audit). Each hop adds 5–15ms; total overhead exceeds 100ms before any business logic runs. Solution: aggregate calls where possible, parallelize independent fetches
  • Consent scope creep (DPDP 2025): the AA consent artifact authorises access to bank statements for underwriting, but a downstream microservice uses the same data for marketing segmentation. Under DPDP 2025 Phase 1 (effective immediately), purpose limitation is enforceable — the gateway must enforce consent scope per downstream service, not just at the entry point
  • Timeout cascade (NPCI 30s): Service A calls Service B (timeout: 10s), which calls Service C (timeout: 10s), which calls NPCI (timeout: 10s). If Service C is slow, Service B waits 10s, then Service A waits 10s for B — total wall time: 20s, leaving only 10s for NPCI response within the 30s session limit. Solution: end-to-end deadline propagation, not per-service timeouts
Synthesis: The API gateway is where RBI compliance is either enforced or leaks. DLG 2025's CIMS portal reporting requirement means every loan origination event must be captured, formatted, and transmitted from the gateway layer — not reconstructed from downstream service logs after the fact. The PM must ensure that compliance middleware (PMLA screening, consent scope enforcement, audit logging, CIMS reporting) runs at the gateway level as a blocking pre-condition, not as an async afterthought. The gateway is not plumbing — it is the regulatory control plane of the NBFC.

3) LLM Mechanics: Tokens, Transformers & Embeddings

Language models convert text to tokens, process them through a transformer, and produce probability distributions over possible next tokens. For NBFC PMs, this matters because: tokens determine cost at Bharat scale, transformers power every credit explanation and UPI dispute resolution, and embeddings drive semantic retrieval of RBI Master Directions and MITC documents. Hindi and regional language queries tokenise differently, affecting cost and quality simultaneously.

  • Tokens are sub-word units (~0.75 words in English; 2–4x more for Hindi/Devanagari) — a 200K token context window holds a complete set of RBI Master Directions for digital lending
  • Transformers use self-attention to weigh every token against every other — long NBFC policy documents in context cost more (attention scales quadratically), requiring careful context budget management
  • Embeddings encode RBI circular clauses as vectors — semantically similar policies cluster together, powering retrieval even when a customer's Hindi query uses different words than the regulation
  • Token cost is your primary AI unit-economics lever at India's scale — Hindi system prompts cost 2–4x more tokens than English equivalents, multiplied by Bharat's high transaction volumes

4) Retrieval-Augmented Generation (RAG)

RAG solves the core NBFC AI problem: models don't know your latest RBI Master Direction, NBFC credit policy, MITC version, or Account Aggregator schema. RAG retrieves the right context at query time and injects it into the prompt, grounding every answer in current regulatory reality. This is the architecture behind every NBFC policy assistant, credit copilot, and compliance Q&A tool — and the mechanism RBI expects when it asks for explainable AI decisions.

  • The ingestion pipeline runs offline: RBI circulars and NBFC policies are chunked, embedded, and stored — its freshness directly determines your RBI compliance posture, not just product quality
  • The query pipeline runs in real time: the customer's query is embedded, matched against stored RBI policy chunks, and top-k clauses are injected into the prompt — grounding the credit decision in current regulatory text
  • Reranking re-scores retrieved RBI circular chunks for relevance — it adds ~30–80ms but is worth it when an incorrect MITC citation is a Digital Lending Guidelines violation
  • Data freshness is a regulatory risk: stale RBI Master Direction chunks produce non-compliant policy answers — your re-ingestion schedule is an RBI compliance commitment, not an engineering detail

RAG failure modes mapped to pipeline stages:
  • Ingestion rot: the RBI repeals DLG 2022 on May 8, 2025 and replaces it with DLG 2025 — but the ingestion pipeline still has the old circular indexed. The RAG system confidently cites repealed regulation. 60% of production RAG deployments now include day-one ingestion freshness monitoring because this is the root cause of most compliance hallucinations
  • Embedding mismatch (Hindi/Hinglish): the embedding model was trained on English text but 60% of customer queries arrive in Hindi or Hinglish via WhatsApp. Retrieval precision drops 30–50% on code-switched queries. Multilingual embedding models (e.g., multilingual-e5) partially address this but require eval on actual NBFC query distribution
  • Retrieval dilution: the query "what is the DLG cap?" retrieves 5 chunks, 3 of which are from the repealed 2022 guidelines and 2 from the 2025 replacement. The LLM averages the conflicting information instead of prioritising the newer source. Metadata filtering (by date, by circular number) is essential, not optional
  • Reranker false confidence: the reranker scores a semantically similar but factually outdated chunk higher than the correct chunk. Rerankers optimise for relevance, not recency or correctness. Post-retrieval date filtering must supplement reranker scores
  • Prompt assembly overflow: 5 retrieved chunks at 800 tokens each = 4,000 tokens of context. Add system prompt (500), user query (100), and MITC template (300) = 4,900 tokens input. On a 4K context model, the last chunk gets truncated — which may be the most relevant one. Context budget management is a PM responsibility
  • Citation hallucination: the LLM generates a response and attributes it to "RBI Circular dated 15.03.2025" — a circular that doesn't exist. Citation verification against the actual retrieved chunks must be an automated eval, not a manual spot-check. RBI FREE-AI's explainability pillar makes verifiable citations the strongest form of AI transparency for regulated entities
Where RAG works for NBFC
  • RBI policy Q&A: well-defined corpus, factual answers, citation-verifiable. The canonical RAG use case for Indian financial services
  • MITC generation: retrieve the correct product-specific MITC template, populate with borrower-specific terms. High-value, high-compliance, and RAG is the right architecture
  • Servicing copilot: customer asks about prepayment charges — RAG retrieves the specific loan product terms and generates a customer-safe response. Bounded scope, verifiable output
  • Compliance audit preparation: retrieve all RBI circulars applicable to a specific NBFC product category. Human reviews the output — RAG accelerates, doesn't replace
  • Internal knowledge base: credit analysts querying policy documents, collections teams looking up restructuring guidelines. Low-stakes, high-volume, clear ROI
Where RAG doesn't work for NBFC
  • Multi-hop regulatory reasoning: "does this MSME borrower qualify for the CGTMSE guarantee given their GST filing history and the latest RBI circular on MSME classification?" — requires 3+ reasoning steps across multiple documents. RAG retrieves documents, it doesn't reason across them
  • Real-time credit decisioning: the credit decision must complete in <200ms. RAG adds 300–800ms of retrieval latency. Use pre-computed BRE rules for real-time decisions, RAG for explanation generation after the fact
  • Numerical analysis: "what is the average NPA rate across all NBFCs with AUM >₹5,000 crore?" — RAG retrieves text, not structured data. Use SQL + analytics, not RAG
  • Adversarial regulatory queries: a user deliberately asks misleading questions to extract incorrect regulatory guidance. RAG's retrieval is indifferent to adversarial intent — the safety layer must sit outside RAG, not inside it
Synthesis: RAG is a pipeline of independently-failing stages — ingestion, embedding, retrieval, reranking, prompt assembly, generation, and citation. Each stage can fail silently, and failures compound. The PM's job is not to "implement RAG" but to own the freshness SLA for ingestion (≤24h after RBI circular publication), the retrieval precision target (>0.85 on golden regulatory queries), and the citation verification rate (100% for customer-facing outputs). RBI FREE-AI's explainability pillar makes RAG citation the strongest available form of AI transparency for regulated entities — but only if the citations are verified, not hallucinated. Own the ingestion pipeline, or discover your compliance gaps in a GRO complaint.

5) Agentic AI Systems

Agents extend LLMs with the ability to plan, use tools, and act across multiple steps autonomously. An NBFC agent might fetch AA-consented bank statements, check a CIBIL score, retrieve relevant RBI policy, call a CBS credit API, and produce a MITC-compliant decision log — without a human in the loop. PMs must understand the reliability math: each step introduces failure probability, and agents multiply these risks. Six steps at 85% reliability each yields only 37% end-to-end success (0.85⁶ ≈ 0.377) — and must complete within NPCI's session timeout.

  • The ReAct loop (Reason + Act) is the core agent pattern: the LLM reasons about which NPCI/AA/CBS tool to call, acts, observes the result, and repeats — until the credit decision is made or a stopping condition is reached
  • Tool calls are deterministic wrappers around NPCI and CBS systems — idempotency keys and maker-checker approval flows are non-negotiable when tools initiate real UPI transactions or NBFC loan disbursements
  • Memory maintains NBFC case state across reasoning steps — AA consent artefact IDs, CIBIL score timestamps, and open policy queries held in context, subject to DPDP Act data retention requirements
  • Human-in-the-loop checkpoints are a PM and RBI compliance decision: under Digital Lending Guidelines 2022, no AI agent may initiate a loan disbursement without a human approval step — write this into the acceptance criteria

11) Portfolio Health & NBFC Economics

VP Product candidates at NBFCs are expected to understand P&L — not just product metrics. NIM, NPA, CRAR, and ALM are the vocabulary of every NBFC board meeting. A PM who proposes a product feature without understanding its impact on credit cost, capital allocation, or ALM mismatch is seen as a risk, not an asset. This section maps the financial and risk management frameworks that shape every NBFC product decision.

Portfolio Health

Portfolio Health Metrics

PM role: own the portfolio quality cockpit — the weekly view that credit, risk, and product review together.

  • DPD Buckets: Days Past Due distribution — DPD 0 (current), DPD 1–30 (early delinquency), DPD 31–90 (sub-standard), DPD 90+ (NPA). Each bucket has a different cost to cure and collection strategy.
  • NPA Stage Classification: Under IndAS 109 — Stage 1 (current, 12-month ECL), Stage 2 (credit deteriorated, lifetime ECL), Stage 3 (defaulted, lifetime ECL). Stage migration rates are a leading indicator of future P&L impact.
  • Vintage Curves: Plot NPA rate by loan origination month at 3, 6, 12, 24 months — reveals whether recent vintages are worse than older ones, signalling underwriting deterioration before it shows in aggregate NPA.
  • Portfolio at Risk (PAR): PAR 30 = (outstanding balance of loans DPD 30+) / total AUM. Industry benchmark varies by product — personal loan PAR 30 target typically < 3%, MSME < 5%.
PM Metric PAR 30 · Stage 2 migration rate · Vintage NPA at 12M · Write-off rate
AI Opportunity EWS models predict Stage 1→2 migration 30–60 days before the DPD clock starts. Vintage analysis automation catches underwriting drift in real time.
Unit Economics

NBFC Unit Economics

PM role: understand how every product decision shifts the P&L — ticket size, tenure, segment, and AI automation all move these levers.

  • Net Interest Margin (NIM): Yield on loans minus cost of borrowing funds. Typical NBFC: yield 18–24%, cost of funds 10–13%, gross NIM 8–12%. AI-powered underwriting reduces credit cost; AI origination automation reduces opex ratio.
  • Credit Cost: Provisioning expense as % of AUM, driven by NPA rate and recovery. Target < 3–5% for personal lending, < 2% for secured. AI EWS reduces credit cost by catching deterioration early.
  • ROA & ROE: Return on Assets (target 2–4%) and Return on Equity (target 12–18%) are the board-level profitability metrics. A PM's product decision is ultimately measured by its impact on ROA.
  • Cost-to-Income: Opex as % of total income. Best-in-class digital NBFCs achieve < 40% via automation. AI reduces cost-to-income through straight-through processing in LOS (target: 70%+ automation rate).
PM Metric NIM · Credit cost · Cost-to-income ratio · ROA

ALM

Asset Liability Management (ALM)

PM role: do not create product features that worsen ALM mismatch without risk committee sign-off.

  • The Mismatch Problem: NBFCs borrow short (6–12M CPs, NCDs, bank credit lines) but lend medium (2–5 year personal loans). This duration mismatch is a structural liquidity risk — what caused the IL&FS crisis in 2018.
  • ALM Buckets: RBI requires quarterly ALM reports showing cash inflows and outflows by time bucket (0–1M, 1–3M, 3–6M, 6M–1Y, 1–3Y, 3–5Y, 5Y+). Negative bucket gaps signal rollover risk.
  • LCR & NSFR: Liquidity Coverage Ratio and Net Stable Funding Ratio — mandated for Upper Layer NBFCs under SBR Feb 2026. Middle/Upper Layer NBFCs must maintain LCR ≥ 100%.
  • PM Implication: Features like moratorium, bullet repayment, or prepayment options directly impact ALM buckets. Every such feature requires a treasury sign-off before product spec finalisation.
PM Metric ALM gap by bucket · LCR (if applicable) · Borrowing mix diversity
RBI / Compliance RBI SBR Framework Feb 2026: Upper Layer NBFCs must file quarterly ALM returns. Negative structural gap beyond regulatory threshold triggers supervisory action.
Capital

Capital Adequacy & SBR

PM role: know which SBR tier your NBFC is in — it determines which products you can build and what governance your product decisions require.

  • CRAR: Capital to Risk-Weighted Assets Ratio — NBFCs must maintain minimum 15% (Tier 1: 10%). Each loan product has a risk weight. High risk-weight loans (unsecured, subprime) consume more capital per rupee disbursed.
  • SBR Tiers (RBI Feb 2026): Base Layer (most NBFCs, lighter touch) → Middle Layer (systemically important, closer oversight) → Upper Layer (~50 largest NBFCs, near-bank regulation) → Top Layer (systemic concern, conversion to bank pathway).
  • PM Implications: Upper Layer NBFCs must have board-level risk committees, independent risk functions, and formal model risk governance — all of which PM must design products around.
  • Co-Lending Impact: Co-lending with banks (RBI Co-Lending Directions Jan 2026) allows NBFCs to originate PSL loans and transfer 80% to bank — freeing up NBFC capital while meeting PSL obligations.
PM Metric CRAR · Tier 1 capital ratio · Risk-weighted asset growth
RBI / Compliance RBI SBR Framework Feb 2026: know your tier. RBI Co-Lending Directions Jan 2026: FLDG/DLG cap at 5% of portfolio covered.
Collections

Collections Funnel & Strategy

PM role: design collections as a data and UX problem — AI improves recovery while keeping the NBFC compliant with RBI fair practice requirements.

  • Collections Funnel: Pre-due (D-5 to D-1) → Soft follow-up (D+1 to D+30) → Field collection (DPD 30–90) → Legal/SARFAESI (DPD 90+) → One-Time Settlement (OTS) → Write-off. Each stage has different cost, recovery probability, and RBI compliance requirements.
  • AI in Collections: Settlement propensity model (which NPA accounts will accept OTS and at what discount), right-party contact prediction (best time and channel per customer), and next-best-action routing (field vs digital vs legal vs OTS).
  • SARFAESI & DRT: For secured loans, SARFAESI Act enables asset repossession without court order (threshold: ₹1 lakh outstanding, NPA classification). Debt Recovery Tribunal (DRT) for larger defaults. These processes take 12–36 months — NPA prevention is far cheaper.
  • OTS Policy: One-Time Settlement is a credit committee decision — AI estimates the settlement amount the customer is likely to accept and what the NBFC's recovery NPV is, enabling structured OTS without per-case credit committee approval for standard buckets.
PM Metric Resolution rate by DPD bucket · Self-cure rate (1–30 DPD) · NPA recovery rate · OTS acceptance rate
RBI / Compliance RBI Guidelines on Fair Practices Code: AI communications cannot threaten, must respect DND hours (8 AM–7 PM only), and must include GRO contact in every message. All AI collection scripts require compliance sign-off.
Fraud & AML

Fraud Detection & AML

PM role: balance fraud prevention with customer experience — a 2% false positive rate on 1 lakh daily applications = 2,000 legitimate customers wrongly declined.

  • Fraud Detection Stack: Device fingerprinting (TrustDecision, IDfy), velocity rules (same PAN on multiple devices), synthetic identity detection (Aadhaar + PAN mismatch + bureau profile inconsistency), CKYC deduplication, and graph-based fraud ring detection.
  • Loan Stacking: A borrower applying simultaneously at multiple NBFCs using the same documents — detectable only via cross-bureau enquiry velocity and CKYC fingerprint matching. A 2–3 day bureau reporting lag is the fraud window.
  • AML / PMLA: Prevention of Money Laundering Act requires NBFCs to file STRs (Suspicious Transaction Reports) within 7 days of suspicion. AI flags unusual disbursement patterns, third-party account transfers, and structuring behaviour.
  • Fraud Loss Budget: Typically 0.1–0.3% of disbursement. AI fraud models are measured on: fraud detection rate (sensitivity), false positive rate (specificity), and cost per fraud case caught. An ensemble model with human review for high-risk cases typically achieves the best P&L outcome.
PM Metric Fraud detection rate · False positive rate · Fraud loss as % of disbursement · STR filing rate
